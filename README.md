## Задание 1. Различия режимов репликации Master-Slave и Master-Master

Главное отличие между схемами заключается в распределении прав на запись данных, топологии потоков синхронизации и логике обеспечения отказоустойчивости.

### 1. Схема Master-Slave (Ведущий — Ведомый)
* **Потоки данных:** Запись (`WRITE`) и изменение данных происходят строго на сервере-источнике (Master). Ведомые серверы (Slaves) работают в режиме `read-only`, непрерывно считывая бинарный лог изменений (`binlog`) ведущего сервера и воспроизводя их у себя.
* **Основное назначение:** Масштабирование операций чтения (клиенты читают со Slave-нод, разгружая центральный узел) и создание изолированных серверов для резервного копирования или аналитики.
* **Отказоустойчивость:** Высокая доступность на чтение. При выходе из строя Master-сервера запись прекращается до тех пор, пока один из Slave-серверов вручную или с помощью оркестратора (например, Orchestrator, MHA) не будет повышен до роли Master.

### 2. Схема Master-Master (Ведущий — Ве ведущий)
* **Потоки данных:** Любой из серверов в кластере полноценно принимает как запросы на чтение (`READ`), так и на запись (`WRITE`). Изменения, совершенные на одной ноде, асинхронно или полусинхронно реплицируются на другую ноду (и наоборот).
* **Основное назначение:** Обеспечение высокой доступности на запись (High Availability), распределение нагрузки между дата-центрами и географическое балансирование.
* **Сложность:** Главный недостаток схемы — риск **конфликтов репликации**, когда на двух узлах параллельно модифицируются одни и те же строки. Для предотвращения пересечения первичных ключей при генерации `AUTO_INCREMENT` требуется обязательная настройка смещений (`offset` и `increment`).

## Задание 2. Конфигурация Master-Slave репликации


### Файлы конфигурации

**`my-master.cnf`**:
```ini
[mysqld]
server-id = 1
log_bin = mysql-bin
binlog_format = ROW
```

**`my-slave.cnf`**:
```ini
[mysqld]
server-id = 2
relay_log = mysql-relay-bin
log_bin = mysql-bin
read_only = 1
```

## Задание 3*. Конфигурация Master-Master репликации

Архитектура была модернизирована в кольцевую топологию, где оба сервера выступают одновременно в роли источников изменений и их приемников.

### Настройка предотвращения конфликтов ID
Чтобы предотвратить пересечение первичных ключей при параллельной записи, конфигурация автоинкремента была изменена.

* **Сервер 1 (`mysql-master`):**
  ```ini
  auto_increment_increment = 2
  auto_increment_offset = 1
  ```

* **Сервер 2 (`mysql-slave`):**
  ```ini
  auto_increment_increment = 2
  auto_increment_offset = 2
  ```
